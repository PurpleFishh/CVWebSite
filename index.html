Beautiful Nightmare

Java

SQLite

/

Description


This is a 2D platformer game made in Java using Java AWT Graphics. The game is designed mainly on an Entity Component System(ECS) and is using a SQLite database for saving the scores and game level state.

Architectural Design


The game architecture revolves around the Entity Component System (ECS), with entities serving as the central building blocks. Entities are essentially empty objects to which components are added to form complete objects. To streamline development, entities are structured hierarchically, inheriting from a base Object class and extending it with necessary components.

These objects are organized within scenes, forming the foundation of the game's scene system. Scenes are managed using a stack-based approach, with the active scene residing at the top of the stack (for example, the pause menu scene takes precedence over the game scene when opened).

Animation is handled through state machines, where each animation represents a state, and transitions are triggered by specific keys. Sprites represent individual frames of animation, collectively forming dynamic sequences. Keyboard and mouse input are managed using the observer pattern, with components subscribing to a global listener. When an event occurs, all subscribed components are notified accordingly.

To facilitate saving and restoring game states, the memento pattern is used, allowing the player and enemyes state to be saved into a SQLite database at any point in the game.

Chicken Quest

C++

/

Description


2D platformer game made in C++ using OOP principles in SDL2 library

About the game


The game is about a chicken that needs to fight the dogs that stole the Magic Chocolate Egg from her Town. You need to beat all the dogs to collect the eggs from every level. Our chicken special ability is that it's eggs explodes.

Architectural Design


The game's architectural design adheres to the Entity Component System (ECS) pattern, offering the flexibility to construct entities to which components can be added. These components function akin to properties for the entity, designed with minimal interdependencies for smoother integration. To streamline ECS usage, entities are instantiated utilizing the Factory design pattern.

Entities in the game are organized within scenes, each scene comprising distinct layers for effective object ordering and management. Scenes are stored in a list, enabling selective rendering and updating of the active scene. Modification of scenes and switching between active scenes can be performed seamlessly at any point.

Entities' colliders are implemented as boxes. For moving objects, collision detection is handled by assessing potential collisions with designated objects, such as map tiles, enemies, or attack damage collision boxes. Collision detection is executed via examination of four collider corners, flagging collisions upon detection.

Map loading is facilitated through file-based parameters, incorporating tile IDs and collision data. This method affords the flexibility to add or modify maps without necessitating alterations to the game's core codebase. Maps are constructed using tiles and exported for integration into the game environment.

DipBet

Java

/

Description


A plugin done for Minecraft Servers in Java using SpigotAPI. It is using the SpigotAPI to work with the server getting information about what is happening in the game and interacting with it.

About the game


DipBet is a betting plugin with the roulette game where the players can bet their money. Participants can choose from three color options - red, black, or green - and specify the desired betting amount. In the event of a correct prediction, players stand to win double their bet for red or black, and a whopping fourteen times their bet for the green color. The game is using a GUI, is fully configurable, has commands and permission. The plugin is published on the Spigot platform available for download

Architectural Design


The plugin is centered around the Game class which manages the roulette system. The configuration offers the flexibility to opt for either a global roulette accessible to all players or individual roulettes for each player. It is implemented in such a way that the differences between the to is the number of instances of Game class, not being dependent on the game mode.

Game states are effectively managed by a state machine, with each state dictating specific behaviors within the game. It can only transition to the next state when it is triggered, so the changing of game behavior being under control.

A Minecraft server is using Spigot to work with plugins, so for interacting with the server the plugin uses SpigotAPI. The plugin work with anouther plugin API called Vault to work with the economy system of the server. Lombok is used for enhanced readability and ease of coding, enabling the generation of getters, setters, and builders through annotations alone. Additionally, serialization and deserialization of items and their properties are facilitated by the XItemStack API from Xseries.

The plugin commands are structured according to the Composite design pattern, facilitating efficient organization and execution. Command hierarchies are stored in a Trie data structure, where leaf nodes correspond to implemented commands, while non-leaf nodes represent command groups. Each node within the Trie is assigned a tag, allowing for the grouping of commands or subgroups under a common identifier.

The plugin is created in such a way that everything can be configurable from the config files. Every aspect, from timers to winning multipliers, GUI appearance to messages and titles, can be modified to suit specific preferences. Every config is a singleton implementing a geneneral config interface, loading the data at initialization or on reload making the acces to datas more facile.
